## 핵심 결정: 비싼 모델을 안 쓰고, 대신 "수요"를 측정하기

가장 고민한 건 추출에 어떤 Gemini 모델을 쓸지였다. 처음엔 당연히 정확도 높은 `gemini-3.5-flash`를 기본으로 박아뒀다. 그런데 타겟을 다시 보니, 대부분의 사용자는 **깨끗하게 잘 찍힌 증명서**를 올린다. 깨끗한 문서엔 굳이 비싼 모델이 필요 없다.

그래서 기본 모델을 저비용·고속인 `gemini-3.1-flash-lite`로 내렸다.

- 대안: `gemini-3.5-flash` 유지 (정확하지만 느리고 비쌈)
- 선택: `flash-lite` 기본 + 고정밀 경로는 함수 인자로만 남김
- 트레이드오프: 흐릿하거나 오래된 문서에선 정확도가 떨어질 수 있음

문제는 "그럼 고정밀이 필요한 사람은 얼마나 될까?"인데, 이건 코드로 추측할 수 없다. 그래서 만들기 전에 **fake-door**로 측정하기로 했다. 추출 화면에 "정밀 분석(준비 중)" 버튼을 두되, 누르면 진짜 3.5 모델이 도는 게 아니라 관심만 1회 기록한다.

```ts
// 정밀 분석 버튼 → 고정밀 추출을 "실행"하지 않고 수요만 카운트
export async function recordFeatureInterest(featureKey: string) {
  const user = await requireUser();
  await db
    .from("feature_interest")
    .insert({ feature_key: featureKey, user_id: user.id })
    .onConflict("feature_key,user_id"); // 같은 사람 중복 클릭은 멱등 처리
  return { ok: true };
}
```

- `unique(feature_key, user_id)`로 distinct 수요만 센다 (한 사람이 10번 눌러도 1명)
- 클릭이 충분히 쌓이면 그때 3.5 경로를 실제로 연결한다
- 안 쌓이면? lite로 충분했다는 뜻이고, 비싼 모델은 안 만든 게 이득이다

기능을 먼저 만들고 수요를 기다리는 대신, 버튼만 만들고 수요를 먼저 확인하는 쪽이 주말 빌드엔 맞다고 느낀다.

## 막혔던 지점: AI 코딩 에이전트가 최신 모델명을 "없는 모델"이라며 옛 버전으로 바꿔놨다

코드를 정리하다가 이상한 걸 발견했다. PLAN에 박아둔 고정밀 모델 `gemini-3.5-flash`가, 어느새 코드에선 `gemini-2.5-flash`로 바뀌어 있었다. 내가 바꾼 기억이 없다.

범인은 빌드를 맡긴 **AI 코딩 에이전트**였다. 자기 학습 데이터에 `gemini-3.5-flash`가 없으니 "존재하지 않는 모델"이라 단정하고, 가장 가까운 옛 버전인 `2.5-flash`로 친절하게(?) 내려놓은 것이다. 무서운 건 **실제 API 에러가 단 한 번도 없었다**는 점이다. 런타임이 멀쩡히 돌아서, 추측으로 바뀐 모델명은 코드 리뷰로 직접 보지 않으면 영영 못 잡는다.

해결 자체는 단순했다 — 웹 검색으로 모델 ID를 직접 확인하니 `gemini-3.5-flash`는 2026-05-19 GA된 정식 stable 모델이었다. 그대로 롤백.

```ts
// AI 에이전트가 "학습 데이터에 없으니 가짜"라며 2.5로 내려둔 걸 되돌림
// 기본은 lite, 고정밀 경로만 3.5-flash (2026-05-19 GA된 정식 ID)
const model = highAccuracy ? "gemini-3.5-flash" : "gemini-3.1-flash-lite";
```

진짜 교훈은 따로 있다 — **AI는 자기 지식 컷오프 이후에 나온 것을 "틀린 것"으로 취급하고 조용히 고친다.** 에러가 없으니 더 위험하다. 그래서 "실제 에러 없이 추측으로 모델명을 바꾸지 말 것"을 아예 프로젝트 철칙으로 박아뒀다. AI에게 코딩을 맡길수록 모델명·API 버전 같은 "최신 사실"은 사람이 다시 봐야 한다.

## 결과

- 주말 이틀, 12개 태스크, TypeScript/TSX 약 2,300줄(43파일)로 배포까지 끝냈다
- AI 추출은 기본 lite 모델 하나로 동작하고, 고정밀은 fake-door 버튼으로 수요만 받는 중이다
- 가입·결제 지표는 아직 0이다. 이제 막 띄웠으니 솔직히 0이 맞다. 앞으로 대기명단 이메일 수와 "정밀 분석" 클릭 수를 보면서 다음 주에 뭘 붙일지 정한다
